# Performance Analyst - パフォーマンス分析スキル

計測データと数値根拠に基づいてボトルネックを特定し、費用対効果の高い最適化を優先度付きで提案します。

## 使用タイミング

- 「パフォーマンスを分析して」「遅い箇所を特定して」
- 「ボトルネックを探して」「処理が重い」「レスポンスが遅い」
- 「メモリ使用量が多い」「CPUが高い」

## 実行フロー

### 1. 問題の特定

現象と計測値を確認：

| 確認項目 | 内容 |
|---------|------|
| 症状 | 遅い処理、メモリリーク、CPU高負荷 |
| 計測値 | 現在の応答時間、メモリ使用量 |
| 目標値 | 目標とする性能指標 |
| 発生条件 | データ量、同時接続数、特定操作 |

### 2. 分析対象の分類

問題の種類を特定し、適切な分析アプローチを選択：

| 分類 | 症状 | 主な原因 |
|------|------|---------|
| **計算量** | データ増加で急激に遅くなる | O(n²)以上のアルゴリズム、冗長ループ |
| **I/O** | DB・ファイル・API処理が遅い | N+1問題、インデックス欠落、不要なリクエスト |
| **メモリ** | 使用量が増加し続ける | オブジェクト保持、クロージャリーク、キャッシュ肥大 |
| **レンダリング** | UI描画が重い | 不要な再レンダリング、大きなDOM操作 |
| **並行処理** | スレッド/プロセスがボトルネック | ロック競合、非効率なasync処理 |

### 3. コード分析

対象コードを読み込み、ボトルネックを特定：

```
【特定された問題】
箇所: ファイル名:行番号
問題: ○○ループ内でDBクエリが発行されている（N+1問題）
計算量: O(n) → データ件数に比例してクエリ数が増加
影響: 100件のデータで100回クエリ発行
```

### 4. プロファイリング手法の提案

言語・フレームワーク別の計測方法を提示：

| 対象 | 計測ツール |
|------|-----------|
| TypeScript/Node.js | `console.time()`, Chrome DevTools, clinic.js |
| Python | `cProfile`, `line_profiler`, `memory_profiler` |
| C#/.NET | BenchmarkDotNet, dotMemory, Visual Studio Profiler |
| SQL | EXPLAIN ANALYZE, スロークエリログ |
| React | React DevTools Profiler, Lighthouse |

### 5. 改善案の優先度付き提示

費用対効果で優先順位を決定：

```
【優先度: 高】N+1問題の解消
- 改善前: ループ内で個別クエリ × n回
- 改善後: JOINまたはIN句で一括取得
- 期待効果: レスポンス時間 500ms → 20ms（96%改善）
- 実装難易度: 低

【優先度: 中】インデックス追加
- 改善前: フルスキャン
- 改善後: インデックス使用
- 期待効果: クエリ時間 200ms → 5ms
- 実装難易度: 低

【優先度: 低】キャッシュ導入
- 期待効果: 繰り返しアクセスの排除
- 実装難易度: 中（キャッシュ戦略の設計が必要）
```

### 6. 改善前後の計測方法の提示

改善効果を定量的に確認する手順を提示：

- ベースライン計測（改善前の数値記録）
- 改善実施
- 同条件での再計測
- 本番環境での監視設定

## 注意事項

- **計測なき最適化はしない**: 推測ではなく計測データに基づく
- **早すぎる最適化を避ける**: 明確なボトルネックのみを対象にする
- **改善前後の比較を必ず行う**: 数値で効果を確認する
