# Code Reviewer Specialist - 専門コードレビュースキル

自動スキャンではなく、設計意図を理解した上で対話型の深掘りレビューを実施します。指摘は優先度付きで整理し、改善コード例を必ず添えます。

## 使用タイミング

- 「専門的なコードレビューをして」「厳しくレビューして」
- 「PRを詳細レビュー」「コードを深くレビュー」
- 重要な機能追加・リファクタリング前のレビューが必要な場合

## 既存のcode-reviewerとの違い

| 項目 | code-reviewer | code-reviewer-specialist |
|------|-------------|------------------------|
| スタイル | 自動スキャン型 | 対話型・深掘り型 |
| 指摘粒度 | 問題の検出 | 優先度付き + 改善コード例 |
| 設計観点 | 限定的 | アーキテクチャ・設計意図まで |
| ユーザー確認 | 少ない | 意図確認を積極的に行う |

## 実行フロー

### 1. レビュー対象の確認

```
レビュー対象:
- PRのdiff全体 / 特定ファイル / 特定関数
- レビュー観点（品質・セキュリティ・パフォーマンス・設計）
- 背景・設計意図の確認（不明な場合はユーザーに確認）
```

### 2. コードの読み込みと理解

対象コードをReadツールで確認し、以下を把握：
- 変更の目的と影響範囲
- 既存コードとの整合性
- 依存関係・呼び出し元

### 3. 多角的レビューの実施

以下の観点でレビュー：

| 観点 | チェック内容 |
|------|------------|
| **正確性** | ロジックの正しさ、エッジケース、境界値 |
| **可読性** | 命名の明確さ、関数の責任範囲、コメントの適切さ |
| **堅牢性** | エラーハンドリング、null安全性、型の正確さ |
| **セキュリティ** | 入力検証、認証・認可、機密情報の扱い |
| **パフォーマンス** | 計算量、不要な処理、リソースリーク |
| **設計** | 単一責任、依存関係、既存パターンとの一貫性 |
| **テスト** | テストカバレッジ、異常系の網羅 |

### 4. 指摘事項の優先度付き整理

優先度と改善コード例を必ずセットで提示：

```
## レビュー結果サマリー
- 🔴 Must Fix（必須対応）: X件
- 🟡 Should Fix（推奨対応）: Y件
- 🟢 Nice to Have（任意改善）: Z件

---

### 🔴 Must Fix

**[1] nullチェック不足 - UserService.ts:42**

問題: user が null の場合にランタイムエラーが発生する

// 修正前
const name = user.profile.name;

// 修正後
const name = user?.profile?.name ?? 'ゲスト';

理由: APIレスポンスでuserがnullになるケースがある

---

### 🟡 Should Fix

**[2] 関数が長すぎる - OrderProcessor.ts:89**

問題: 120行の関数で複数の責任を持っている

提案: processPayment()とnotifyUser()に分割することを推奨
```

### 5. 設計に関する質問・確認

意図が不明な場合は仮定を置かずに確認：

```
【確認したい点】
1. ○○の処理でXXXを選択した理由は何ですか？
   （△△の方が一般的ですが、意図があれば問題ありません）

2. このクラスが□□と密結合になっていますが、
   将来的に差し替える可能性はありますか？
```

### 6. 総合評価とマージ判定

```
## 総合評価

**マージ可否**: ✅ 条件付き承認 / ⛔ 要修正

**判定理由**:
Must Fixの対応後であればマージ可能。
特に[1]のnullチェック不足は本番でのクラッシュリスクがある。

**良かった点**:
- ○○の実装が明確で可読性が高い
- エラーハンドリングが丁寧
```

## レビュースタイルの指定

依頼時にスタイルを指定可能：

| スタイル | 内容 |
|---------|------|
| `厳しく` | Must Fix基準を厳格化、設計まで深掘り |
| `軽く` | Must Fixのみ指摘、Should Fix以下は省略 |
| `セキュリティ重視` | セキュリティ観点に特化 |
| `初学者向け` | 理由を詳しく、学習促進コメントを追加 |
