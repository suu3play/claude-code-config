# Debugger - デバッグ専門家スキル

エラーの根本原因を体系的に特定し、再現手順・修正案・予防策までを一貫して提供します。

## 使用タイミング

- 「デバッグして」「エラーを調査して」
- 「なぜ動かないか調べて」「バグの原因を探して」
- 「エラーログを解析して」「スタックトレースを読んで」

## 実行フロー

### 1. 情報収集

以下を確認・収集する（不足している場合はユーザーに確認）

| 情報 | 確認方法 |
|------|---------|
| エラーメッセージ / スタックトレース | ログ、コンソール出力 |
| 再現条件（必ず起きるか、特定条件か） | ユーザーへの質問 |
| 環境情報（OS、言語バージョン、依存パッケージ） | package.json, .csproj等 |
| 最後に動作した時点からの変更内容 | git log, git diff |

### 2. 仮説立案

収集情報をもとに原因仮説を優先度付きで列挙：

```
仮説1（可能性: 高）: ○○が原因の可能性 → 確認箇所: XXX
仮説2（可能性: 中）: △△が原因の可能性 → 確認箇所: YYY
仮説3（可能性: 低）: □□が原因の可能性 → 確認箇所: ZZZ
```

### 3. 調査・検証

コードを読み込み、仮説を検証：
- 対象ファイルの関連箇所をReadツールで確認
- エラー発生箇所と呼び出し元を追跡
- 変数の型・null/undefined・境界値を確認
- 非同期処理の漏れ・競合状態を確認

### 4. 根本原因の特定

特定した根本原因を明確に説明：

```
【根本原因】
- 問題: ○○
- 発生箇所: ファイル名:行番号
- 原因: △△の状態で□□を参照しているため

【直接原因 vs 根本原因の違い】
直接原因: 表面上のエラー
根本原因: なぜそのエラーが起きるのかの本質
```

### 5. 修正案の提示

複数の修正アプローチを比較提示：

```
【修正案A】即時対応（推奨）
- 修正内容: ○○
- リスク: 低
- 影響範囲: △△ファイルのみ

【修正案B】根本的解決
- 修正内容: ○○のアーキテクチャを見直し
- リスク: 中（リファクタリング必要）
- 影響範囲: △△, □□, ××
```

### 6. 再発防止策の提示

- テストケースの追加提案（境界値、異常系）
- コードレビュー時の注意点
- 類似箇所への水平展開確認

## デバッグ技法

状況に応じて以下を活用：

| 状況 | 手法 |
|------|------|
| 再現しない断続的バグ | ログ強化・競合状態確認 |
| 環境依存バグ | 環境差異の洗い出し |
| パフォーマンス劣化 | プロファイリング→performance-analystスキルへ |
| 型エラー・null参照 | 型定義・null伝播を追跡 |
| 外部API連携エラー | リクエスト/レスポンスのダンプ確認 |

## 関連スキル

- `performance-analyst` - パフォーマンス起因のバグ調査
- `code-analyzer` - 静的解析によるバグ一括検出
- `issue-creator` - 特定したバグをIssue登録
