# Dev Planner - 改修前設計ドキュメント作成スキル

改修作業前に必要な設計ドキュメントを作成し、全体仕様書（SPEC.md）を最新状態に保ちます。

## 使用タイミング

- 「設計書を作成してから開発して」
- 「要件定義して」「ドキュメントを作ってから改修して」
- 「仕様書を更新して」「SPEC.mdを更新して」

## 実行フロー

### 1. プロジェクト分析

プロジェクトの特性を自動検出:

| 検出項目 | 判定方法 | スキップ対象 |
|---------|---------|------------|
| DB利用有無 | `.sql`, `migrations/`, `schema.prisma`, `models/`等の有無 | DB設計書 |
| フロントエンド有無 | `pages/`, `views/`, `components/`, `*.xaml`, `*.html`等の有無 | 画面遷移図 |
| 改修規模 | 変更ファイル数・影響範囲の見積もり | 小規模は確認 |

### 2. 改修規模判定

**小規模と判定する条件（いずれかに該当）**:
- 変更予定ファイルが5件未満
- 新機能追加がない（既存機能の修正のみ）
- UI変更・DB変更が伴わない

小規模の場合は、「ドキュメントを作成しますか？（y/n）」と確認する。

### 3. ドキュメントフォルダ作成

作成する場合、次の連番フォルダを `docs/revisions/` に作成:

```
docs/
├── SPEC.md                           # 全体仕様書（常に最新）
└── revisions/
    ├── 001_YYYYMMDD_[改修名]/        # 最初の改修
    │   ├── 要件定義.md
    │   ├── DB設計.md                 # DBあり時のみ
    │   └── 画面遷移図.md             # フロントエンドあり時のみ
    └── 002_YYYYMMDD_[改修名]/        # 次の改修
```

**命名規則**:
- 連番: 既存フォルダ数+1（3桁ゼロ埋め: 001, 002...）
- 日付: 今日の日付（YYYYMMDD形式）
- 改修名: 依頼内容から3〜5語で命名（例: `user-auth-feature`, `payment-bug-fix`）

### 4. ドキュメント作成

作成対象のドキュメントをTemplateに従い作成:

| ドキュメント | テンプレート | 作成条件 |
|------------|-----------|---------|
| 要件定義.md | `references/templates.md` § 要件定義 | 常に作成 |
| DB設計.md | `references/templates.md` § DB設計 | DBあり時のみ |
| 画面遷移図.md | `references/templates.md` § 画面遷移図 | フロントエンドあり時のみ |

### 5. SPEC.md 更新

`docs/SPEC.md`が存在しない場合は新規作成、存在する場合は変更履歴と影響セクションを更新:

- **新規作成**: `references/templates.md` § 全体仕様書 のテンプレートを使用
- **更新**: 「変更履歴」に今回の改修を追記、影響を受けるセクションを更新

### 6. 開発モード確認

ドキュメント作成完了後、開発モードをユーザーに確認する:

**質問**: 「開発を開始します。一気通貫モード（全ファイルを自動承認で一括作成）で進めますか？」

| 選択肢 | 説明 | 対応 |
|-------|-----|-----|
| 通常モード | 各ファイル作成時に承認プロンプトが表示される | そのまま開発開始 |
| 一気通貫モード | `--dangerously-skip-permissions` フラグを使用 | 案内文を表示して開発開始 |

**一気通貫モードを選択した場合の案内**:
```
一気通貫モードで進めます。
現在のセッションではすでに承認権限が設定されているため、
または次回から以下のコマンドで起動してください:
  claude --dangerously-skip-permissions
全ファイルを連続で作成します。完了後にまとめてレビューしてください。
```

### 7. 完了報告

作成したドキュメント一覧と次のアクション（開発作業開始）を報告。

## スキップロジック詳細

### DBなしプロジェクトの検出パターン
以下がすべて存在しない場合はDBなしと判定:
- `*.sql`, `*.db`, `*.sqlite`
- `migrations/`, `prisma/`, `alembic/`
- `models.py`, `schema.prisma`, `*Context.cs` (DbContextを含む)
- `app/models/`, `db/`, `database/`

### バックエンドのみの検出パターン
以下がすべて存在しない場合はフロントエンドなしと判定:
- `src/components/`, `src/pages/`, `src/views/`
- `*.xaml`, `*.html` (テンプレート用途)
- `resources/views/`, `templates/`
- `public/index.html`, `index.tsx`, `App.tsx`

## 使用例

**ユーザー**: ユーザー認証機能を追加したい。設計書を作ってから開発して

**動作**:
1. プロジェクト分析 → DBあり、フロントエンドあり、大規模
2. `docs/revisions/001_20260226_user-auth-feature/`を作成
3. 要件定義.md、DB設計.md、画面遷移図.md を作成
4. `docs/SPEC.md`を新規作成（初回）
5. 作成完了を報告 → 開発作業の準備完了を提示

---

**ユーザー**: バグ修正して（小規模・DB変更なし・画面変更なし）

**動作**:
1. プロジェクト分析 → 小規模と判定
2. 「設計ドキュメントを作成しますか？（スキップも可）」と確認
3. 不要と回答 → ドキュメント作成スキップ、SPEC.mdの変更履歴のみ更新

## 注意事項

1. **SPEC.mdは必ず更新**: 改修のたびに変更履歴セクションを更新する
2. **連番は自動計算**: 既存 `docs/revisions/` フォルダ数+1で決定
3. **テンプレート参照**: 各ドキュメントは必ず`references/templates.md`のテンプレートに従う
4. **Mermaid記法**: ER図・画面遷移図はMermaid記法で記述（VSCode/GitHubでプレビュー可能）

## リファレンス

- **各ドキュメントテンプレート**: `references/templates.md` - 要件定義・DB設計・画面遷移図・全体仕様書のテンプレート
